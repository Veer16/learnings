Tier 1 in Power Automate — step by step, and why each step exists

01

Prerequisites & the one thing that surprises people

  • A Power Automate environment you can build in (the default environment is fine to start).
  • A SharePoint site where you can create a list (a team site you own).
  • Permission to create a Microsoft Form (any M365 user).
  • Three approver mailboxes/groups: the manager is resolved automatically; HR and Protective Services should each be a mail-enabled group (not a single person) so cover and turnover don't break the flow.
  • A service/shared account to own the flow — strongly recommended so the automation isn't tied to your personal identity (and doesn't break when someone leaves).
The Dataverse gotcha The Approvals connector stores its records in Microsoft Dataverse. The first time anyone runs an approval flow in an environment, the system auto-provisions a Dataverse database, and the person who runs that first flow needs an environment admin role. After that, other makers don't need elevated rights. It can take a few minutes the first run. This is included with your licence — it isn't a premium/paid component — but plan for it so the first test run doesn't fail mysteriously.
02

Three design decisions, made up front

1 · Sequential approvals, not parallel

Route Manager → HR → Protective Services in sequence. If the manager declines, HR and Protective Services are never disturbed, and the chain mirrors how authority actually flows. (Parallel is faster but bothers everyone at once and is harder to reason about for an access-control decision.)

2 · "Create an approval" + "Wait for an approval", not "Start and wait"

"Start and wait for an approval" is the one-action shortcut, but it does not expose the Approval ID, which you need to send reminders or cancel a pending request. Splitting into Create an approval (which returns the Approval ID) and Wait for an approval unlocks the reminder/escalation pattern in §07.

Shortcut option If you want the very fastest first cut, you can use Start and wait for an approval per stage with a Timeout set in the action's Settings, and a "run after → has timed out" branch for escalation. It's simpler but gives you no mid-wait reminders. Start there if you're time-boxed, then upgrade to the split pattern.

3 · Approval type "First to respond" with a single assignee per stage

Each stage has one decision-maker (or one group acting as one). Use Approve/Reject – First to respond so the first response from the assigned group settles that stage. ("Everyone must approve" is for when every named person must individually sign off — not what you want when HR is a shared queue.)

Build the Microsoft Form

Keep it short — every field you ask for is a field you must validate. Restrict the form to your organisation so it records the responder's identity (you'll use that as the requester, and to resolve the manager).

06

The flow, action by action

Build an automated cloud flow. Each step below names the exact connector and action, what to set, and what it teaches.

1

Trigger on form submission

Microsoft Forms · When a new response is submitted

Select your form. The trigger fires once per submission but only gives you a response ID, not the answers.

TeachesForms triggers are intentionally lightweight — they hand you a pointer, and you fetch the content in the next step. This keeps the trigger fast and lets one flow serve high volumes.
2

Fetch the answers

Microsoft Forms · Get response details

Pass the Response Id from the trigger. You now have the location, dates, justification, and — because the form is org-restricted — the Responder's email.

Teaches"Get response details" is the partner to every Forms trigger. The Responder's email token is your anchor identity for everything downstream.
3

Resolve the requester & their manager from the directory

Office 365 Users · Get user profile (V2) Office 365 Users · Get manager (V2)

Feed the responder's email into Get user profile (V2) (gives display name, UPN, department) and into Get manager (V2) (gives the manager's email and name). Use the manager's Mail for the first approval stage.

Handle the nullIf a user has no manager set in Entra, Get manager (V2) fails. Wrap it in a scope or add a "configure run after → has failed" branch that routes to a fallback approver (e.g. a managers' distribution group) and flags the record, rather than letting the whole flow error out.
TeachesDirectory-sourced approvers are the single biggest reliability win in Tier 1 — you've eliminated mistyped names and stale manager data in one move.
4

Create the record

SharePoint · Create item

Write all captured fields into the list with Status = Submitted. Keep the returned ID and Link to item — you'll reference both in updates and approval cards.

TeachesPersisting state immediately means that even if a later step fails, the request is never lost — you have a row to inspect and resume from.
5

Validate before bothering any approver

Control · Condition (or Scope + Condition)

Check: both dates present, start not in the past, end after start, and duration within policy (see §08 for the exact expressions). Compute DurationDays and store it.

If invalid: update Status = Returned-Invalid, send the requester a specific "here's what to fix" email, then Control · Terminate with status Cancelled.

TeachesValidation-first is the whole point of Tier 1. Catching bad requests here is what ends the back-and-forth your team does today — garbage never enters the approval chain.
6

Gate 1 — Manager approval

Approvals · Create an approval  →  Wait for an approval

Update Status = Pending Manager. Then Create an approval:

  • Approval type: Approve/Reject – First to respond
  • Assigned to: the manager's email from step 3
  • Title: e.g. Overseas work — [RequesterName] — [Location]
  • Details: a markdown summary (dates, duration, justification)
  • Item link: the SharePoint item link from step 4

Capture the Approval ID, then add Wait for an approval referencing it (the reminder loop in §07 sits alongside this). On completion, branch on Outcome:

  • Approve → write ManagerDecision = Approve + comments, continue to Gate 2.
  • Reject → Status = Rejected, notify requester with the comment, Terminate.
TeachesThe Outcome and Responses tokens carry the decision, the approver, the comment and the timestamp. Writing them to the list at each gate is how you build an audit trail with zero extra effort.
7

Gate 2 & Gate 3 — HR, then Protective Services

Approvals · Create an approval  →  Wait for an approval  (×2)

Repeat the Gate 1 pattern, assigning to the HR group then the Protective Services group, updating Status to Pending HR and Pending ProtSvc as you enter each. Same approve/reject branching; same write-back of decision + comment.

TeachesAssigning to a group (with First-to-respond) means anyone on the HR/Protective Services queue can action it — resilient to leave and turnover, unlike naming an individual.
8

All approved — the clean handoff

SharePoint · Update item Teams · Post message Office 365 Outlook · Send an email (V2)

Set Status = Approved-Ready and stamp ApprovedOn. Post a message to your team's Teams channel and email the team queue with the validated summary and the list-item link. Email the requester that it's approved and being provisioned.

TeachesThis single notification is what fixes your visibility problem. Your team stops watching a silent queue and instead receives one fully-validated, fully-approved record — the manual PIM step that remains is now a one-minute action with no ambiguity.
07

The reminder & escalation pattern

This is the piece that stops approvals sitting for days. It lives inside each gate, running beside the "Wait for an approval".

After Create an approval, add a Control · Parallel branch so two things run at once:

  1. Branch A — Wait for an approval (referencing the Approval ID). This completes the moment the approver responds.
  2. Branch B — the reminder loop: a Do until that exits when the approval is responded to, containing a Delay (e.g. 2 days) followed by a reminder email/Teams nudge to the approver.

For hard escalation, give the Wait for an approval a Timeout in its Settings (ISO 8601, e.g. P3D = 3 days). Add a branch with Configure run after → "has timed out" that reassigns to a fallback approver (a delegate or the manager's manager) and flags the record.

Hard platform limit Power Automate approvals time out after 30 days — long-running approvals beyond that need a restart pattern (re-create the approval and resume). For this process, keep stage timeouts short (2–3 business days) so you never approach that ceiling, and escalation handles the stragglers.
Do-until tuning A naïve Do-until reminder waits the full Delay even after the approver responds, and can fire a final reminder late. Set the loop's exit condition to check the approval's response state, and keep the Delay modest. If this gets fiddly, ship the simpler "Timeout + escalate on timed-out" version first and add reminders later.
08

Validation expressions you can paste

Use these in the Condition's expression editor. They compare dates by ticks (100-nanosecond units), which avoids string/locale pitfalls. Replace StartDate/EndDate with your Forms tokens.

End date must be after start date

greater(ticks(EndDate), ticks(StartDate))   // true = valid

Start date not in the past

greaterOrEquals(ticks(StartDate), ticks(startOfDay(utcNow())))

Duration in days, and a 90-day policy cap

// days between the two dates
div(sub(ticks(EndDate), ticks(StartDate)), 864000000000)

// cap check — true = within policy
lessOrEquals(
  div(sub(ticks(EndDate), ticks(StartDate)), 864000000000),
  90
)

A required field isn't empty

not(empty(trim(Justification)))
Timezone & the boundary Forms date answers and utcNow() are UTC; you're in NZ (UTC+12/+13). For "not in the past" that rarely matters, but when these dates later feed the PIM endDateTime (Tier 2), decide whether the window ends at NZ midnight or UTC midnight and convert with convertTimeZone consistently. Document the choice so Tier 2 inherits it.
TeachesTicks-based math sidesteps the single most common Power Automate bug — comparing dates as strings and getting locale-dependent results. 864000000000 is the number of ticks in one day.
09

Test plan & go-live checklist

  • First run as env admin so the Approvals Dataverse DB provisions cleanly.
  • Happy path: valid request → all three approve → record reaches Approved-Ready and the team is notified.
  • Invalid dates: end before start, and a start in the past → both returned to requester with reasons, no approver disturbed.
  • Over-cap duration: > 90 days → returned as invalid.
  • Rejection at each gate: confirm the flow stops, status is correct, requester is told why.
  • No manager in Entra: confirm the fallback branch routes the approval and flags the record (doesn't error).
  • Reminder & timeout: shorten the Delay/Timeout to minutes for testing, confirm a reminder fires and that timeout escalates.
  • Notifications: requester and team messages contain the right links and read clearly.
  • Ownership: flow is owned by the service/shared account, with a co-owner so it's not a single point of failure.
10

Gotchas worth knowing now

  • Dataverse provisioning needs an env admin on the first-ever approval run in that environment.
  • "Start and wait" hides the Approval ID — use "Create an approval" + "Wait for an approval" if you want reminders or to cancel a request.
  • 30-day approval ceiling — keep stage timeouts short; escalate stragglers rather than letting them run for weeks.
  • Get manager (V2) throws on null — always have a fallback approver branch.
  • Date comparisons — compare by ticks, not as strings; mind the NZ↔UTC offset, especially where these dates feed Tier 2's PIM window.
  • Group approvers, not individuals — assign HR and Protective Services to mail-enabled groups with First-to-respond so the process survives leave and turnover.
  • Connector check before you call this "free" — keep every action on the standard connectors listed at the top. If a future change adds an HTTP/Graph/ServiceNow action, that's the Tier 2 jump and brings the premium licensing discussed in the architecture doc.
