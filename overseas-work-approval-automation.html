Bottom line first

Build Tier 2: Forms → Power Automate Approvals → Graph (PIM for Groups)

Keep the request, the four approvals, validation, reminders and notifications entirely on standard connectors you already own (Forms, Approvals, SharePoint, Teams, Outlook). Add one premium component — a single call to Microsoft Graph — that writes the user into the overseas group as an active, time-bound PIM assignment.

That one call carries the approved start and end dates, so the platform removes access automatically when the period ends. The single biggest risk in your current process — forgetting to revoke access — stops being a task at all.

If HNZ holds (or will buy) Entra ID Governance licences, also evaluate Entitlement Management access packages as a near-zero-code alternative (Tier 3) — it does request, multi-stage approval, time-boxing and recertification out of the box.

02

The current process, and exactly where it leaks

The process splits cleanly into two halves: a cross-functional approval phase you don't control, and a technical provisioning phase your team runs by hand. Both halves leak, but for different reasons.

Where time and trust are lost

  1. Paper-then-scan intake. A physical form is signed by four parties and scanned. Nothing is validated until it reaches you — so incomplete or wrongly-dated requests are only caught at the end.
  2. Approvals collected off-system. Manager, HR and Protective Services sign sequentially with no tracking, no reminders, no escalation. This is where most of the elapsed time disappears.
  3. Queue with no signal. Requests land in ServiceNow but your team isn't pinged, so they sit. This is a notification gap, not a ServiceNow limitation.
  4. Manual verification. Re-reading the scan to confirm all four signatures, mandatory fields and date logic — repeated for every request.
  5. Privilege elevation by hand. Sign in → activate the Device Management role via PIM → find the overseas group.
  6. Manual PIM assignment. Add the user, switch eligible→active, type in start and end dates, save, double-check.
  7. Expiry depends on memory. If dates are mistyped or the assignment is set without an end, access lingers — the highest-impact failure mode.
Root causes, not symptoms The pain isn't "too many steps" — it's that validation happens last, approvals are untracked, and revocation is a human action. A good design moves validation to the front, makes approvals first-class and auditable, and makes revocation automatic. Most of your listed challenges collapse once those three things are true.
03

The insight that changes the whole design

You're already using PIM for Groups for this group. PIM can create an active assignment that is time-bound — with a start time and an explicit end time — and it will deprovision the membership itself when that end time passes.

That capability is reachable from a single Microsoft Graph call. So the automation doesn't need a separate "scheduled clean-up job" to strip access later, and it doesn't need anyone to remember anything. It makes one correct request at approval time, and PIM owns the rest of the lifecycle.

Why this matters "Failure to remove access when the period expires" — your most serious risk — is solved natively by the platform, not by code you have to maintain. Every other tier in this document inherits this property for free.

One caveat worth recording: if the tenant's Entra ID P2 / Governance licensing ever lapses, existing time-bound assignments convert to permanent (they stop expiring). Treat the P2/Governance entitlement as a hard dependency of the control, not just a nice-to-have.

Target end-to-end architecture

A single orchestration with one front door, sequential approvals with timeouts, validation gates on both sides of the approvals, a privileged provisioning step isolated behind app-only auth, and an audit ledger.

How each requirement is met

Your requirementHow the target meets it
Validate required fieldsFlow validates on submit and again before provisioning; bad requests are returned to the employee, never reaching your team.
Approval routingSequential Approvals actions: Manager → HR → Protective Services, each actionable in Teams, Outlook or the Approvals app.
Reminders & escalationPer-stage timeout; on timeout the flow re-sends, then escalates/reassigns to a delegate or manager's manager.
Group membership assignmentOne Graph call writes an active PIM-for-Groups assignment — no interactive elevation by your team.
Automatic removal at expiryThe assignment carries an end date; PIM deprovisions it automatically. No clean-up job.
Stakeholder notificationsTeams/Outlook messages at submit, each decision, completion and pre-expiry.
Audit trailApprovals history + Entra audit logs + a state row stamped with the ticket number tie the whole chain together.

Three solution tiers

Same backbone, increasing automation and governance. Each tier is a viable stopping point — Tier 1 alone removes most of the daily pain.

Tier 1

Quick win — automate intake & approvals, keep provisioning manual

Effort: days
Cost: $0 extra

Forms + Power Automate Approvals + a SharePoint list + Teams/Outlook notifications. The flow validates fields, routes the four approvals, chases and escalates, then hands your team a single fully-approved, pre-validated record (and pings them). Your team still does the PIM assignment — but now it's a one-minute, zero-ambiguity step.

Pros

  • Buildable in days; nothing new to buy
  • Kills the visibility, chasing and missing-info problems immediately
  • All standard connectors — no premium licence, no app registration
  • Low risk; easy to get governance sign-off

Cons

  • Provisioning & PIM elevation still manual
  • Residual human-error risk at the assignment step
  • Doesn't yet exploit native auto-expiry
ComplexityLow
LicensingExisting M365 (E3/E5) only
SecurityNo new privileged identity; team still elevates via PIM as today
MaintenanceMinimal — one flow, one list
Tier 2 · recommended

Automated provisioning via Graph + PIM for Groups

Effort: 1–2 wks
Cost: low

Tier 1 plus the Graph call. On full approval the flow posts an active, time-bound assignment to the overseas group using app-only auth (dedicated Entra app, secret in Key Vault, least-privilege Graph permission). PIM handles start activation and end-date removal. Stamp the ServiceNow/record number into the request for traceability.

Pros

  • Near-full automation; your team approves exceptions only
  • Native auto-expiry — the revocation risk disappears
  • No standing human privilege; the elevation step is gone
  • Strong, queryable audit via Entra logs

Cons

  • One premium component (the HTTP/Graph call) needs licensing
  • An app registration holds a privileged Graph permission — needs review & secret hygiene
  • Slightly more to build & test (auth, throttling, error paths)
ComplexityMedium
LicensingM365 + one premium path (see §08); P2/Governance for assigned users (already in place)
SecurityApp-only SP, Key Vault secret, least-privilege scope, conditional-access exclusions managed
MaintenanceSecret/cert rotation; monitor Graph version & throttling
Tier 3

Enterprise — buy-not-build with Entitlement Management, or full custom platform

Effort: weeks
Cost: higher

Two flavours, depending on appetite:

3a — Entitlement Management access packages (least code). An access package fronts the request, runs multi-stage approval, grants time-bound membership of the overseas group, auto-expires, and supports access reviews for recertification — mostly configuration, not code. Trade-off: the approver model is more prescriptive (manager / nominated approvers / sponsors), and it needs Entra ID Governance licences.

3b — Full custom platform. Dataverse as system of record, provisioning moved to a Logic App (Standard) or Azure Function with managed identity, Application Insights / Log Analytics telemetry, a Power BI dashboard, dead-letter handling and retry policies, plus Access Reviews and optionally Lifecycle Workflows.

Pros

  • 3a: most governance for the least custom code
  • Recertification, separation of duties, rich reporting
  • Scales to other time-bound access scenarios
  • 3b: full observability and control of the provisioning path

Cons

  • Highest licensing (Governance and/or Dataverse)
  • 3a: approval routing less flexible than custom Approvals
  • 3b: most components to build, secure and operate
ComplexityHigh
LicensingEntra ID Governance; Dataverse / Logic Apps / Azure as used
SecurityManaged identity (no secrets), access reviews, full audit, SoD
MaintenanceMore surface area; offset in 3a by Microsoft-managed lifecycle
How to choose Ship Tier 1 first regardless — it's the fastest relief and de-risks everything downstream. Promote to Tier 2 once the app registration and licensing are signed off; that's the recommended steady state. Reach for Tier 3a if HNZ already invests in Entra ID Governance and prefers configuration over custom flows.

Should ServiceNow stay in the loop?

Yes — but as an audit/ITSM ledger, not as the orchestrator or the place work waits. Today ServiceNow is where requests stall; in the target it's where the completed transaction is recorded.

Role for ServiceNowVerdict
Orchestrator (drives approvals/provisioning)Remove — Power Automation does this better and is where your team already lives.
Intake (the request front door)Optional — fine if you prefer staff to raise it as a catalog item; otherwise Forms is lighter.
Audit / ITSM recordKeep. The flow creates/updates a record so existing ITSM reporting and retention still apply. Stamp the record number into the PIM request's ticketInfo to link both worlds.

The ServiceNow connector is premium, so if you keep the writeback it sits in the same licensing bucket as the Graph call (§08) — no separate cost once you've licensed one premium path.

07

Security & governance

Validating approvals you can trust

Replace ink signatures with identity-bound decisions. The Approvals action records who approved, when, and from an authenticated session — far stronger evidence than a scanned signature. Enforce that each stage is actioned by the right principal (e.g. resolve the manager from Entra rather than free-text), and require the approver set to match policy before provisioning proceeds.

Least privilege

  • No standing human admin for this task. The interactive PIM elevation your team does today is replaced by an app-only service principal scoped to exactly one capability.
  • Minimal Graph permission. Grant only what writes a PIM-for-Groups assignment (PrivilegedAccess.ReadWrite.AzureADGroup) — confirm the exact least-privilege scope against current Graph docs at build time rather than over-granting directory roles.
  • Secret hygiene. Client secret/cert in Key Vault; prefer a managed identity if you host the call in a Logic App or Function (Tier 3b) so there's no secret at all.
  • Scoped group. The SP can only touch the overseas group, not arbitrary privileged groups.

Audit & compliance

  • Entra audit logs capture every assignment immutably — export to Log Analytics for long retention if your policy needs it.
  • Approvals history preserves the decision chain; the state row preserves field-level data and timestamps.
  • Access reviews (Governance) can periodically recertify who's in the overseas group — useful evidence in a healthcare compliance context.
  • Separation of duties: the requester can't approve; approver stages are distinct roles; provisioning is performed by a non-human identity.
Governance dependency The auto-expiry control relies on the tenant retaining Entra ID P2 / Governance licensing for assigned users. Document this as a control dependency — if licensing lapses, time-bound assignments silently become permanent.

Licensing & cost

The architecture is deliberately shaped so that almost everything runs on connectors already covered by your M365 entitlement. Cost concentrates in exactly one place: the privileged Graph call.

Cheapest way to cover the one premium leg

Three options, lowest-friction first:

  • Host the Graph call in an Azure Function or Logic App (Consumption) and have the standard flow call it. This sidesteps Power Automate premium entirely; at this request volume the Azure bill is effectively rounding error, and you get managed identity (no secret). Usually the most cost-effective.
  • A Power Automate per-flow / Process licence attached to the single provisioning flow covers all users of that flow for a flat monthly fee — clean when you'd rather keep everything in Power Automate.
  • A Power Automate Premium per-user licence on the flow's owner/service account — simplest to reason about for a single low-volume automation, billed per user.
On the figures Microsoft renamed and re-priced Power Platform plans through 2025–2026 (the per-app plan was retired, per-flow became "Process", and some seeded credits change later in 2026). Treat any list price as indicative and confirm current NZ pricing via your EA/CSP before committing — the architecture doesn't change, only which line item you choose.
09

Operations: errors, failures, logging, scale

Error handling & approval failures

  • Validation gates on both sides of approvals; invalid or declined requests return to the employee with a clear reason and never reach provisioning.
  • Approval timeouts per stage drive reminders, then escalation/reassignment — overdue approvals can't stall silently.
  • Scope/Try-Catch pattern in the flow: wrap provisioning in a scope with a configured run-after "failed" branch that alerts your team in Teams and writes the failure to the audit record.

API failures & idempotency

  • Throttling (HTTP 429): honour Retry-After with exponential back-off; Graph PIM endpoints are rate-limited.
  • Pre-flight check: use the request's isValidationOnly mode to test the assignment before committing, surfacing policy/MFA blockers early.
  • Idempotency: before posting, check for an existing active assignment for that user+group to avoid duplicates on retries.
  • Token handling: cache the app-only token to its expiry; handle 401 by refreshing once before failing.

Logging, monitoring, reporting

  • Run history in Power Automate for day-to-day triage; pipe failures to a monitored Teams channel.
  • Entra audit logs → Log Analytics for durable, queryable history and alerting on anomalies.
  • Dashboard: Power BI over the state list/Dataverse — requests in flight, time-in-stage, approvals SLA, upcoming expiries, exceptions.

Future scalability

The pattern — validated request → multi-stage approval → app-only, time-bound grant → auto-expiry → audit — generalises to any temporary-access scenario (contractor access, temporary elevated roles, project groups). Parameterise the target group and approver chain and the same engine serves many request types. Moving the provisioning leg to Logic Apps/Functions (Tier 3b) gives you independent scaling and full telemetry as volume grows.

Technical appendix

The one call that does the work

On full approval, the flow (or the Function it calls) posts an active, time-bound membership. startDateTime activates access; expiration with an endDateTime makes PIM remove it automatically. ticketInfo links the ITSM record.

# POST against Microsoft Graph — PIM for Groups
POST https://graph.microsoft.com/v1.0/identityGovernance/privilegedAccess/group/assignmentScheduleRequests
Authorization: Bearer <app-only token>
Content-Type: application/json

{
  "action":      "adminAssign",
  "accessId":    "member",
  "groupId":     "<overseas-workers-group-id>",
  "principalId": "<approved-user-object-id>",
  "justification": "Overseas work — approved request <ref>",
  "scheduleInfo": {
    "startDateTime": "2026-07-01T00:00:00Z",
    "expiration": {
      "type":        "afterDateTime",
      "endDateTime": "2026-09-30T23:59:59Z"
    }
  },
  "ticketInfo": {
    "ticketNumber": "<ServiceNow ref>",
    "ticketSystem": "ServiceNow"
  }
}
Read this line twice Because expiration.endDateTime is the approved overseas end date, no removal logic is ever written or maintained. Set the dates correctly once and the platform owns the rest of the lifecycle. Set "isValidationOnly": true first to dry-run the request before committing.

Build order

  1. Stand up Tier 1 (Forms + Approvals + list + notifications) and run it live — immediate relief.
  2. Register the Entra app, grant the least-privilege Graph permission, store the secret in Key Vault (or use a managed identity via a Function).
  3. Add the validated Graph call behind a Try-Catch scope; dry-run with isValidationOnly.
  4. Wire ServiceNow writeback and the Power BI dashboard.
  5. Layer access reviews / recertification; evaluate Entitlement Management as the longer-term home (Tier 3a).

Prepared as an options analysis for the HNZ overseas-work access process.
